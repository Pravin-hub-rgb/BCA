  • When we talk about cybersecurity, one of the most important things to understand is how systems are attacked — these methods are known as attack vectors. In simple terms, an attack vector is the path or technique used by a hacker to break into a system or cause harm. In this unit, we’ll explore different categories of attack vectors that every IT or security student should know.
  • We’ll start with Buffer Overflow Attacks, which are more technical and involve manipulating memory to crash a system or take control of it — things like stack overflows, string overflows, and even heap or integer overflows fall under this.
  • Then we’ll move into Internal Attacks, which are all about threats that come from within the organization or through common user channels — like suspicious emails, mobile apps, instant messengers, and even physical methods like dumpster diving or shoulder surfing.
  • Finally, we’ll cover Denial of Service (DoS) Attacks, which are designed to overwhelm or crash systems so that real users can't access them. This includes notorious attacks like Ping of Death, SYN flooding, and even modern Distributed DoS (DDoS) methods that use multiple devices to attack at once.

Buffer Overflow Attacks

Buffer overflow attacks are a classic and dangerous class of vulnerabilities that exploit the way memory is handled in low-level languages like C and C++. These attacks occur when a program writes more data to a buffer (like an array or string) than it was designed to hold, causing unexpected behavior. But why is this dangerous? Because memory in programs is structured in predictable ways, overflowing one area can let an attacker take control over critical parts of a program — even injecting and executing malicious code.

Stack Overflows

The stack is a special region of memory where local variables and function calls are stored. It follows a last-in, first-out (LIFO) order. A stack overflow happens when we push more data onto the stack than it can hold. This is the most commonly exploited type of buffer overflow.

  • Attackers often target the return address stored on the stack.
  • By overflowing a local buffer (like a character array), they overwrite the return address with the location of their own malicious code.
  • When the function tries to return, it jumps to the attacker's code instead of the intended return point.

This kind of exploit can give attackers full control of the system if successful. It's one of the reasons functions like gets() or strcpy() are considered dangerous — they don’t check input size.

String Overflows

String overflows are a special case of buffer overflows where unbounded string input causes the overflow. They often happen when dealing with C-style strings (null-terminated character arrays).

  • If a function doesn't limit how many characters it copies or reads into a buffer, it can write past the end of that buffer.
  • This may overwrite adjacent memory, causing crashes, corrupted data, or exploitable conditions — just like stack overflows.

Many famous attacks — like the Morris Worm — used string overflows to spread and execute code remotely. It's why functions like strncpy() and input validation are important.

Heap Overflows

Unlike the stack, which handles temporary and scoped data, the heap is used for dynamically allocated memory (via malloc() or new). A heap overflow happens when data written to the heap overruns the boundary of its allocated space.

  • Heap overflows don't target the return address but can corrupt heap metadata or function pointers.
  • Attackers may exploit this to execute arbitrary code or manipulate program behavior.

These attacks are harder to perform than stack overflows, but they're equally dangerous and harder to detect. Secure memory management and modern heap protection techniques (like heap canaries) help reduce this risk.

Integer Overflows

Integer overflows happen when a calculation results in a number too large (or too small) to be stored in the intended data type. While this isn’t a buffer overflow by itself, it can lead to one.

  • Imagine allocating a buffer using the result of an integer multiplication, but the value overflows and becomes very small.
  • The program then allocates a small buffer but proceeds to copy a large amount of data into it — causing a buffer overflow.

Integer overflows are often overlooked but serve as a stepping stone for more severe attacks, especially in systems where input sizes are calculated without proper checks.

Understanding all these types of overflows is critical for both attackers and defenders. Developers must validate input sizes, use safe libraries, and leverage memory protection features in modern systems to prevent such vulnerabilities.

How to Protect Against Buffer Overflow Attacks

Developers use several tricks and tools to stop buffer overflow attacks from messing with their programs. Here are some common ways they stay safe:

  • Write Code Carefully : Always check the size of the input, set boundaries, and use safer functions when dealing with strings. Basically, don’t trust user input blindly!
  • Use Memory Safety Features : Features like ASLR (Address Space Layout Randomization) help make memory locations unpredictable. Also, marking parts of memory (like the stack or heap) as non-executable stops attackers from running malicious code there.
  • Use Code Checking Tools : There are tools that scan your code (both while writing and during testing) to catch any buffer overflow issues before they become a problem.
  • Run Code in a Safe Zone (Sandboxing) : If a program does get attacked, running it in a restricted space means it can’t do much harm. Also, separating user permissions keeps sensitive stuff safe.
  • Watch for Weird Behavior at Runtime : Some tools keep an eye on how a program behaves while it’s running. If anything looks suspicious (like memory being accessed in a strange way), it can stop the program right away.

Real-World Examples and Case Studies

These vulnerabilities aren't just theoretical. Over the years, real-world incidents have shown how deadly buffer overflow attacks can be. Here's a look at some major examples across each category:

  • Stack Overflow – The Morris Worm (1988): One of the first and most famous worms spread across the internet using a stack buffer overflow in the finger service on Unix systems. It exploited poor input handling to execute remote code.
  • String Overflow – Microsoft IIS Unicode Bug (2001): This vulnerability allowed attackers to send a long string in a web request, exploiting unvalidated Unicode input and crashing or compromising the server.
  • Heap Overflow – Microsoft Windows GDI+ (2004): A heap-based overflow in image parsing routines (used when opening JPG files) could be triggered just by getting a user to open a crafted image, leading to remote code execution.
  • Integer Overflow – Apple's "goto fail" Bug & Adobe Flash: Multiple integer overflow bugs have been found in Apple and Adobe products. In Adobe Flash, such overflows allowed memory mismanagement and code injection through malformed video files.

These examples show that buffer overflow attacks affect every type of system — from web servers and desktop applications to operating systems. Developers must stay vigilant, and secure coding practices must always include bounds checking, safe memory handling, and modern protection mechanisms like ASLR (Address Space Layout Randomization), DEP (Data Execution Prevention), and stack canaries.

Internal Attacks

When we think about cyberattacks, it's easy to imagine hackers breaking in from the outside. But many serious security breaches come from within — through people or systems already inside the organization. These are called internal attacks. They’re dangerous because the attacker often has legitimate access and is trusted. Let’s explore how everyday tools and behaviors — like emails, mobile devices, and even trash bins — can become sources of internal threats.

Emails

Emails are one of the most commonly used communication tools in organizations — and one of the most exploited. An insider with malicious intent can misuse email in various ways, from sending sensitive data out of the company to spreading malware internally.

  • An employee might attach confidential files and forward them to a competitor or personal email.
  • Even well-meaning employees can fall victim to phishing emails that appear internal but are actually crafted by attackers to steal login credentials.

This is why email monitoring, employee training, and data loss prevention (DLP) tools are critical components of internal security.

Mobile Phones

Almost everyone carries a smartphone into the workplace. While convenient, mobile phones can also be a source of data leakage or unauthorized recording.

  • Insiders can use their phones to take pictures of confidential documents or screens.
  • They might connect their phones to company computers and transfer sensitive data using USB tethering or Bluetooth.
  • Even apps on a personal phone can silently record audio or log keystrokes if permissions are misused.

This is why many companies enforce mobile device management (MDM) policies and restrict phone usage in high-security zones.

Instant Messengers

Tools like Slack, WhatsApp, or Microsoft Teams are great for quick communication, but they can also become unmonitored channels for data leakage.

  • An insider can easily copy and paste confidential data into a private message or group chat.
  • Some attackers use messenger tools to coordinate with others during an internal breach.

Even seemingly harmless jokes or links shared in a company chat could lead to social engineering or malware if not properly monitored.

FTP Uploads

FTP (File Transfer Protocol) is used to move large files across systems. It’s fast, but it’s also risky if not secured — especially when internal users have access.

  • A rogue employee might upload confidential documents to an external FTP server they control.
  • If FTP accounts are not properly monitored or logged, such activity can go undetected for long periods.

Using secure alternatives like SFTP, coupled with access control and logging, can help prevent misuse.

Dumpster Diving

Not all internal threats are digital. Dumpster diving refers to the act of digging through physical trash to find sensitive documents or hardware that was improperly discarded.

  • An attacker could recover printed emails, old reports, or USB drives thrown into the bin without shredding or wiping.
  • Even things like sticky notes with passwords or access codes can be valuable to someone digging through the trash.

Proper document disposal policies and shred bins are simple but effective countermeasures.

Shoulder Surfing

Sometimes, all it takes to breach a system is a good set of eyes. Shoulder surfing involves watching someone enter passwords, read confidential screens, or type sensitive information — often without them noticing.

  • This can happen in open office layouts, cafes, elevators, or even during meetings.
  • Insiders might memorize access credentials or take photos discreetly.

Privacy filters, awareness training, and auto-lock policies for unattended computers help reduce this threat.

Bringing It All Together

Internal attacks don’t always look like "hacks" — they’re often subtle, low-tech, and driven by everyday tools. What makes them especially dangerous is the attacker’s access and knowledge of the system. Whether it's through emails, devices, messengers, or even physical observation, understanding these internal threats helps organizations build layered defenses. Combining technology with employee awareness is the key to keeping internal security strong.

Understanding DoS (Denial of Service) Attacks

Imagine trying to enter a shop, but someone keeps blocking the doorway so customers can’t get in. That’s what a Denial of Service (DoS) attack does to websites or online services — it overwhelms them with fake traffic or corrupted requests so that real users can’t use them.

DoS attacks come in different flavors, and each one targets the system in a unique way. Let’s walk through the most common types and see how they work.

Ping of Death

This attack sends oversized or malformed packets using the ping command. Normally, a computer can handle pings easily. But when it's flooded with these abnormal packets, it gets confused and crashes or freezes. It’s like someone sending you a puzzle with missing pieces — over and over again — until your brain gives up!

Teardrop Attack

In this attack, the data packets sent to a computer are broken into fragments — but not properly. When the system tries to put them back together, it fails and may crash. Think of trying to solve a jigsaw puzzle with wrongly shaped pieces that can’t fit — frustrating, right?

SYN Flooding

This one takes advantage of how two computers start a conversation using a process called a "three-way handshake." The attacker floods the target with half-finished connections, never completing the handshake. This leaves the server hanging and eventually runs out of space to handle real users.

  • Step 1: Attacker sends a SYN request (says "Hi!")
  • Step 2: Server replies with SYN-ACK ("Hi back!")
  • Step 3: But the attacker never replies with ACK — leaving the server waiting

This keeps happening until the server is too busy to talk to anyone else.

Land Attacks

A land attack sends a fake request to the server where the source and destination IP addresses are the same — it tricks the system into talking to itself endlessly. It’s like calling your own number and getting stuck in a loop with yourself!

Smurf Attacks

This one is clever. The attacker spoofs (fakes) their IP address and sends a ping request to many devices, asking them to reply to the victim’s IP. Suddenly, the victim is flooded with replies from hundreds or thousands of devices.

  • The attacker sends an ICMP (ping) request to a broadcast address.
  • All devices on that network reply to the victim’s IP.
  • The victim gets overwhelmed with responses they didn’t ask for.

UDP Flooding

Here, the attacker bombards the target with tons of UDP packets on random ports. The system keeps looking for applications to handle these messages — and when it can’t find them, it replies with errors, which eats up resources fast.

Hybrid DoS Attacks

Some attackers mix multiple types of attacks at once — maybe combining SYN flooding with a ping flood or mixing low and high-level techniques. These are harder to detect because they use more than one strategy to break the system.

Application-Specific Attacks

These target particular applications like a web server, database, or login page. Instead of attacking the whole network, they hit the part of the app that does the most work — making it slow or completely unusable.

Distributed Denial of Service (DDoS) Attacks

DDoS is like a mega version of DoS. Instead of one attacker, many devices (often hacked ones called a “botnet”) work together to flood the target. It’s like hundreds of people trying to enter a small store all at once — no one else can get in, and the system collapses.

  • These attacks are usually launched from thousands of infected computers.
  • They can take down large websites, online stores, and even government services.

Defending against DoS and DDoS attacks is super important because these attacks don’t just crash websites — they cost money, damage trust, and sometimes even threaten critical services.
